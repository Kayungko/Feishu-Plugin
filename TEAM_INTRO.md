# Feishu Workspace 插件试用介绍

这是一份比较简短的说明，主要是想邀请大家小范围试用一下 `Feishu Workspace` 这个 Codex 插件。

它不是一个完整产品，也不是想替代大家平时对飞书内容的判断。更准确地说，它是一个把 Codex 和飞书日常办公操作接起来的小工具：让 Codex 在处理文档、Wiki、表格、会议纪要、任务草稿、日程等内容时，少一点重复查询和手工整理。

目前阶段更适合先作为试用版使用。如果过程中遇到识别不准、回复太长、链接打不开、权限不足等问题，都可以直接反馈，我们再根据真实使用情况继续调整。

## 这个插件主要解决什么

平时我们会有一些重复但又不太值得手动慢慢整理的工作，例如：

- 读一篇飞书文档，整理重点、待办、风险和需要确认的问题。
- 把 Wiki 或文档改成更适合分享的口语稿。
- 把会议纪要里的行动项拆成任务草稿。
- 把一批飞书链接整理成资料索引。
- 写表格或创建任务前，先生成一份预览，确认后再继续。
- 查询今天的日程，并整理需要提前准备的事情。

这些事情本身不复杂，但比较耗时间。这个插件希望先把这类低风险、可复查、可撤回的步骤做顺一点。

## 适合什么场景

| 场景 | 是否适合 | 说明 |
| --- | --- | --- |
| 读取文档并总结重点 | 适合 | 适合做初步摘要、待办提取、风险梳理。重要内容仍建议人工复核。 |
| Wiki / 文档整理成分享稿 | 适合 | 适合把书面内容转成更口语化的讲稿或提纲。 |
| 会议纪要提取行动项 | 适合 | 适合先生成任务草稿，不建议直接批量创建任务。 |
| 多个飞书链接整理成资料索引 | 适合 | 适合做资料归类、标题摘要、阅读顺序建议。 |
| 表格写入前做字段预览 | 适合 | 适合先看字段映射、行数和写入影响，再决定是否执行。 |
| 查询个人日程和待办 | 适合 | 适合整理今天或本周要处理的事情。 |
| 直接替你做业务判断 | 不太适合 | 插件只能辅助整理信息，不能替代业务 owner 的判断。 |
| 大批量修改共享文档或表格 | 谨慎使用 | 必须先预览，确认目标、内容和影响范围后再执行。 |
| 发送飞书消息、邮件、创建任务 | 谨慎使用 | 默认只生成草稿，发送或创建前需要明确确认。 |
| 涉及敏感权限、删除、移动文件 | 暂不建议 | 这类操作风险较高，不作为第一轮试用重点。 |
| 依赖飞书项目完整读写 | 视情况 | Project 相关能力依赖额外配置和接口支持，稳定性要单独确认。 |

## 目前已经有的能力

### 文档和 Wiki

- 读取飞书文档或 Wiki。
- 总结背景、结论、待办、风险和问题。
- 把文档改写成口语化分享稿。
- 对文档做“去 AI 味”改写草稿。
- 创建或更新文档前生成写入预览。

### 会议和任务

- 从会议纪要或逐字稿中提取行动项。
- 识别负责人、截止时间、优先级线索。
- 生成飞书任务草稿。
- 创建任务前必须先预览，不会默认直接创建。

### 表格和资料整理

- 把资源路径、案例清单、文本列表整理成表格字段。
- 写入表格或多维表前预览字段映射、行数和影响范围。
- 把多个飞书链接整理成资料索引。

### 日程和协作

- 查询今日日程。
- 整理需要准备的事项。
- 根据文档生成飞书消息草稿或邮件草稿。
- 发送前需要人工确认完整内容。

### 环境和排障

- 检查 `lark-cli` 是否可用。
- 检查当前飞书登录状态。
- 检查插件包脚本是否完整。
- 对常见错误做简单归类，例如登录、权限、scope、配置缺失。

## 安全规则

这个插件目前默认遵守几个规则：

1. 读取类操作可以直接执行。
2. 写入共享资源前必须先给预览。
3. 发消息、发邮件、建任务、邀请参会人前必须确认。
4. 不默认展示内部 skill 列表、长 scope、完整命令日志。
5. 出错时优先说明原因和下一步，而不是贴一大段日志。

如果你发现它没有按这些规则执行，可以直接反馈，这类问题优先修。

## 在 Claude Code 桌面端使用

同一个包现在也支持 Claude Code 桌面端，和 Codex 共用同一套能力、脚本和安全规则，不需要装两份。

接入方式（团队）：

```text
/plugin marketplace add G:\feishu_CLIforMe\feishu-workspace
/plugin install feishu-workspace@feishu-workspace-market
```

装好后，除了直接用自然语言（`feishu-workspace` 技能会按上下文自动触发），还可以用斜杠命令做显式入口：

- `/feishu-workspace:feishu-help` — 介绍能做什么 + 常用提示词
- `/feishu-workspace:feishu-read <飞书链接>` — 读取并总结飞书内容
- `/feishu-workspace:feishu-agenda [日期]` — 今日日程和要准备的事
- `/feishu-workspace:feishu-actions <纪要/链接>` — 提取行动项、生成任务草稿（不直接创建）
- `/feishu-workspace:feishu-doc <内容>` — 整理成飞书文档（写入前先预览）
- `/feishu-workspace:feishu-check` — 检查环境和插件包

下面这些以 `@feishu-workspace` 开头的提示词，在 Claude Code 里去掉 `@feishu-workspace` 直接说，或改用上面对应的斜杠命令即可，效果一样。

## 反馈时可以看这几类问题

试用时不需要特意写很正式的反馈，能说明问题就行。比较希望大家关注这些点：

- 提示词是否自然，还是必须写得很像命令。
- 回复是否太长，是否需要更直接。
- 文档总结有没有遗漏重点。
- 会议行动项有没有误判负责人或截止时间。
- 写入预览是否说清了目标、内容和影响。
- 飞书链接是否能正确识别。
- 报错时是否知道下一步该怎么处理。

## 当前边界

这个插件目前还是团队内部试用版，能力边界需要说清楚：

- 它更擅长整理和转写，不擅长替你做最终判断。
- 它依赖当前账号的飞书权限，没权限的文档读不到。
- 它依赖 `lark-cli` 和对应 scope，环境没配好时会失败。
- 飞书 Project 相关能力还需要额外配置，不建议作为第一轮试用重点。
- 真实写入操作需要谨慎，尤其是共享文档、表格、任务、消息和日程。

## 建议试用方式

可以先找 2 到 3 位同事试用一两天。每个人挑自己真实工作里的一两个低风险场景，不需要刻意构造复杂案例。

第一轮目标不是证明它有多强，而是确认它在日常工作里是否省事、是否稳定、是否容易理解。如果它在某些场景里帮不上忙，也可以直接记录下来，这对后续判断要不要继续投入更有价值。
